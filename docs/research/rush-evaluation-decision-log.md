# Rush.js Evaluation — Multi-Agent Decision Log

**Date:** 2026-06-27  
**Scope:** Evaluate Rush.js as monorepo build/install manager vs ModMe's current stack (Turbo, Bun, Yarn, esbuild, Elysia, NixOS, worktrees)  
**Disposition:** **REJECT** full Rush adoption — **APPROVED** Turbo-first improvement path  
**Audit lever:** `node scripts/research/monorepo-tool-audit.mjs --md`

---

## Understanding Lock

ModMe is a **meta-monorepo**: one Git root orchestrating **independent** Turborepos with **forbidden** cross-stack `workspace:*` links.

| Layer | Tool | Location |
|-------|------|----------|
| Orchestration | Yarn 3.3 | Root `package.json` |
| Primary apps | Turbo + Bun | `next-forge/` (`bun.lock`, `turbo.json`) |
| Legacy agent stack | Turbo + Yarn 3 | `GenerativeUI_monorepo/` |
| Bundling (root scripts) | esbuild | `esbuild.config.mjs` |
| Bun API | Elysia | `next-forge/apps/elysia/` |
| Multi-agent isolation | Git worktrees | `scripts/new-agent-worktree.ps1` |

**Success criteria for this review:** Determine whether Rush improves build/install/CI enough to justify migration cost, without breaking Bun-first next-forge or dual-stack boundaries.

## Phase 1 — Primary Designer (Rush capabilities summary)

### Rush core model

- **Explicit project inventory** in `rush.json` (no wildcard discovery)
- **Single centralized shrinkwrap** per install graph (`common/config/rush/`)
- **Pinned package manager** — Rush installs its own copy of pnpm, npm, or Yarn Classic
- **Commands:** `rush update`, `rush install`, `rush build` (incremental), `rush rebuild` (full)
- **Recommended PM:** pnpm (phantom-dep protection, `strictPeerDependencies`)

### Advanced features researched

| Feature | Purpose | Prerequisite |
|---------|---------|--------------|
| [Incremental builds](https://rushjs.io/pages/advanced/incremental_builds/) | Skip unchanged projects via hash analysis | Default on `rush build` |
| [Build cache](https://rushjs.io/pages/maintainer/build_cache/) | Tarball restore local + Azure/S3 | `build-cache.json`, per-project `rush-project.json` |
| [Phased builds](https://rushjs.io/pages/maintainer/phased_builds/) | `_phase:build` / `_phase:test` parallelism | Build cache required |
| [Cobuilds](https://rushjs.io/pages/maintainer/cobuilds/) | Multi-machine CI via Redis + shared cache | Build cache + Redis |
| [Subspaces](https://rushjs.io/pages/advanced/subspaces/) | Multiple **pnpm** lockfiles in one Rush repo | pnpm 8.7.6+, Rush 5.122+ |
| [Autoinstallers](https://rushjs.io/pages/maintainer/autoinstallers/) | Isolated tool deps for hooks/plugins | `common/autoinstallers/` |
| [Rush plugins](https://rushjs.io/pages/maintainer/using_rush_plugins/) | Extend Rush (cache providers, cobuild) | Autoinstaller + `rush-plugins.json` |
| [CI bootstrap](https://rushjs.io/pages/maintainer/enabling_ci_builds/) | `install-run-rush.js` pins Rush version | Generated by `rush update` |
| [Sparo](https://tiktok.github.io/sparo/) | Sparse checkout profiles for large Rush repos | Rush + PNPM; `sparo clone` required |

### Rush vs Turbo (layering)

| Concern | Rush | Turbo (current) |
|---------|------|-----------------|
| Install governance | Primary strength | Delegates to workspace PM |
| Task graph / caching | Build cache (experimental) + incremental | First-class remote cache |
| PM flexibility | pnpm/npm/yarn only — **no Bun** | Bun, Yarn, pnpm, npm |
| Discovery | Manual `rush.json` entries | `workspaces` in `package.json` |
| Enterprise release | Change logs, version policies, publish | Lighter; Vercel/Next integration |

## Phase 2 — Structured Review

### Skeptic / Challenger

**Assume Rush adoption fails in production. Why?**

1. **Bun blocker:** next-forge is `packageManager: bun@1.3.10` with `bun.lock`. Rush has no `bunVersion` ([issue #4457](https://github.com/microsoft/rushstack/issues/4457)). Elysia builds use `bun build --target bun`.
2. **Dual orchestrator:** Two `turbo.json` files already work. Rush adds `common/config/` ceremony without replacing Turbo's task model — you'd run both or rip out Turbo.
3. **Subspaces trap:** Subspaces enable `workspace:` across subspaces — opposite of ModMe's intentional firewall between stacks.
4. **Migration cliff:** No `rush.json` exists. Adoption = re-register every package, rewrite CI (`verify-forge-ci.ps1`), retrain agents, conflict with upstream next-forge template.
5. **Windows friction:** Team already documents `gh aw compile` hangs on native PowerShell; Rush's PNPM symlink model adds phantom-dep migration pain on Windows.
6. **Cache misconfiguration risk:** Rush docs warn incorrect `rush-project.json` output folders cause silent wrong builds — worse under cobuilds.

**Objections raised:** 6 critical, 0 dismissed without evidence.

### Constraint Guardian

| Constraint | Rush compliance |
|------------|-----------------|
| Bun in next-forge | **FAIL** — no supported path |
| Separate lockfiles (root Yarn, forge Bun, GenUI Yarn) | **FAIL** — subspaces = pnpm only |
| `monorepo-boundaries.mdc` (no cross-stack workspace) | **FAIL** — Rush unifies one workspace |
| Multi-agent worktrees | **NEUTRAL** — worktrees work with any tool; Rush doesn't improve them |
| Cloud Supabase / Prisma deploy order | **NEUTRAL** — orthogonal |
| GitHub Actions path filters | **ALREADY SOLVED** — `dorny/paths-filter` in `ci.yml` |
| NixOS reproducibility | **COMPATIBLE** — Nix pins toolchain; `install-run-rush.js` is complementary *if* Rush existed |
| esbuild root bundling | **COMPATIBLE** — Rush would wrap scripts, not replace esbuild |
| CI time (no remote cache today) | **GAP** — fix via Turbo cache, not Rush migration |

**Verdict:** Rush violates three hard architectural constraints.

### User Advocate

**Developer experience impact:**

- **Negative:** New mental model (`rush update` vs `bun install`), Heft-oriented docs, loss of `bunx` ergonomics, upstream next-forge drift.
- **Positive (if on Rush):** Stricter dependency consistency (`ensureConsistentVersions`), unified change-log workflow — valuable at 100+ packages, not at ~30.
- **Agent UX:** AGENTS.md, worktree doctor, and `run-forge-bun.ps1` are tuned for Turbo+Bun. Rush would obsolete documented paths.
- **What users actually need:** Faster CI, reliable `yarn dev:forge`, clear boundary rules — achievable without Rush.

**Verdict:** Rush solves enterprise governance problems ModMe doesn't have yet; it worsens daily Bun/Elysia workflow.

## Phase 3 — Integrator / Arbiter

### Objection resolutions

| Objection | Decision | Rationale |
|-----------|----------|-----------|
| Bun unsupported | **ACCEPTED** | Hard blocker; community uses Turbo+Bun for Elysia |
| Subspaces unify dual-stack | **REJECTED** as solution | Current boundary rules + separate lockfiles are the correct abstraction |
| Rush build cache > Turbo | **REJECTED** | Turbo remote cache is S-effort; Rush cache is XL + experimental |
| Sparo for clone speed | **REJECTED** for now | Requires Rush; path filters + worktrees already address pain |
| Cobuilds for CI scale | **REJECTED** | ~31 packages; single-runner GHA sufficient |
| Nix + Rush bootstrap | **REJECTED** | Nix should wrap current toolchain, not hypothetical Rush |
| Keep `.cursor/rules/rush.mdc` as reference | **ACCEPTED** | Documentation only; no repo adoption |

### Approved alternative path (S–M effort)

1. **Turbo remote cache** on `next-forge` CI (Vercel free tier or self-hosted)
2. **GHA `actions/cache`** for `.turbo` on GenerativeUI job (pattern exists in UniversalWorkbench copies)
3. **Keep** `dorny/paths-filter` — equivalent selective CI without Sparo
4. **Keep** `.githooks/pre-commit` + `pre-commit-checks.mjs` — already solves hook-when-install-broken
5. **Optional:** Nix flake pinning Node/Bun/Yarn for reproducible dev shells (complements Turbo)
6. **Align** `catalog-ci.yml` with `ci.yml` working-directory patterns

### When to revisit Rush

Only if ALL become true:

- GenerativeUI sunset complete
- next-forge voluntarily migrates Bun → pnpm
- Package count / release governance needs exceed Turbo's scope
- Team accepts single `rush.json` workspace
